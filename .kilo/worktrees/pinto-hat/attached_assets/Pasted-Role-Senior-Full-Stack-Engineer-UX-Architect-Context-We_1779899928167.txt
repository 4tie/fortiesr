Role: Senior Full-Stack Engineer & UX Architect.

Context: We need to vastly improve application reliability, user transparency, and debugging capabilities. This involves adding robust global error handling, an interactive "Thinking" state in the Chat Panel, and diagnostic test buttons in the Settings UI for both API keys and selected models.

Task: Implement the following four structural UI/UX upgrades without breaking the "Cold Black" theme.

### 1. Robust Global Error Handling (Toasts)
- Implement a unified error-catching mechanism on the frontend using DaisyUI Toast components (positioned at the bottom-right or top-center).
- Whenever any backend request fails (e.g., timeout, rate limit, invalid OpenRouter key, or Python subprocess crash), intercept the error and display a clear, styled error toast (e.g., soft red background) rather than failing silently.

### 2. Interactive "Thinking" State & Logs (Chat Panel)
- When the AI is processing a request, display a sleek, pulsing animation in the chat feed (e.g., pulsing dots or a glowing brain icon).
- Make this "Thinking..." indicator **clickable/expandable** (using an accordion or `<details>` HTML structure).
- **Inside the expanded view:** Display the underlying thought process. If the model provides `reasoning` tokens via OpenRouter (e.g., DeepSeek R1), stream them here in a muted gray font. If the AI agent is running internal tools, display a live log of steps (e.g., "1. Fetching profile...", "2. Generating strategy code...", "3. Validating JSON...").

### 3. API Key Diagnostic Button (Settings Tab)
- Add a "Test API Key" button immediately below or beside the OpenRouter API Key input field.
- Wire it to a lightweight backend route (e.g., fetching account limits or a simple auth ping to OpenRouter).
- Show a loading spinner inside the button when clicked, followed by a clear visual indicator: a Green Badge ("Valid Key") or a Red Badge ("Invalid/Unauthorized").

### 4. Model Diagnostic Buttons (Settings Tab)
- Add a "Test Model" button underneath both the 'Execution Model' and 'Thinking Model' dropdowns.
- Wire these to send a minimal prompt (e.g., "Respond with 'OK'") to the specifically selected model via OpenRouter.
- Display an inline visual status (Green dot for "Active/Working", Red dot for "Failing/Rate-Limited") so the user knows the model is ready before starting complex tasks.

Ensure all new components handle asynchronous loading states gracefully and maintain the sleek, dark aesthetic of the application.
