1. Strategy Selection
Decision: Use AdaptiveFactory as specified in your auto_quant_options.json.

Why: Testing with the actual production-intended strategy structure ensures that our Phase 3 and Phase 5 regex/string injections target the correct class properties and methods.

2. Data Availability
Decision: Let the pipeline's Auto-Download feature handle it.

Why: This is the perfect opportunity to stress-test your newly minted Phase 1 Data Healer. If there are data gaps or missing pairs between 20250101 and 20260601, we want to watch the subprocess spin up, fetch the data, and stream the status via WebSockets.

3. Freqtrade Installation & Path
Decision: Point the pipeline to your local virtual environment binary (typically venv/bin/freqtrade or .venv/Scripts/freqtrade.exe on Windows). If Freqtrade is completely isolated or missing from the testing env, use a Python-based stub/mock that mimics Freqtrade's JSON output structures.

Why: We want to test the orchestrator's logic, dictionary parsing, and state machine transitions. A stub allows us to validate the code without environmental blockages.

4. Execution Method
Decision: Run via a standalone test script (e.g., test_e2e_pipeline.py).

Why: Bypassing the FastAPI layer removes network overhead and keeps the logs isolated. It allows the AI assistant to easily read stack traces, catch unhandled exceptions, and modify the core pipeline code in a clean bubble.

5. Runtime Constraints (Hyperopt Epochs)
Decision: Absolutely scale it down. Drop the hyperopt epochs from 160 to 3 to 5 epochs for the test run.

Why: We only need to verify that the pipeline correctly invokes the process, parses the resulting epoch dictionaries, runs the WFA segments, and aggregates weights. Running 160 epochs will just melt your CPU for hours without adding any QA value.

6. Expected Output Location
Decision: Yes, saving to user_data/auto_quant/{run_id}/{StrategyName}_Optimized.py is correct.

Why: Keeping optimized strategies compartmentalized by run_id prevents historical runs from overwriting each other.

7. Error Handling Scope
Decision: Focus exclusively on pipeline module code bugs, regex/parsing logic, and configuration anomalies.

Why: If the AI hits a syntax error in your orchestrator or an unhandled KeyError during weight normalization, it should refactor the code and fix it. If it encounters a dropped internet connection or missing system dependencies, it should gracefully log a clean, non-crashing pipeline failure.
