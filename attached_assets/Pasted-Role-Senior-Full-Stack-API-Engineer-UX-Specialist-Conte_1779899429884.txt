Role: Senior Full-Stack API Engineer & UX Specialist.

Context: We need to bulletproof our AI Model Selection in the Settings UI. Currently, users can accidentally select models that do not support tool/function calling for the Execution agent, causing silent failures. The list of models must also be strictly free and updated dynamically from OpenRouter.

Task: Implement dynamic OpenRouter API fetching, strict capability filtering, and visual UI indicators in the Settings component.

### 1. Live OpenRouter API Integration (Backend)
- Create a new utility function `fetch_openrouter_models()` that makes a REST call to `https://openrouter.ai/api/v1/models`.
- **Primary Filter:** Strictly filter the API response to ONLY keep models that are 100% FREE (where both prompt and completion pricing equal 0 or it contains "free" in the ID).

### 2. Execution Model Dropdown (Strict Tool-Calling Only)
- In the Settings UI for "Execution Model", map only the filtered free models that are known to support Tool/Function Calling. 
- *Logic:* Since OpenRouter's API might not explicitly list a boolean for tool-calling, implement a robust regex or keyword filter on the model IDs/descriptions (e.g., allowing models like `meta-llama/llama-3...`, `google/gemini...`, `mistralai/...`, or specifically identifying instruct/tool-capable tags) to ensure the dropdown ONLY contains safe, tool-execution-ready models.

### 3. Thinking Model Dropdown (Visual Indicators)
- For the "Thinking Model" dropdown, allow ALL free models from the dynamic list.
- **Visual Enhancement:** Append a Brain icon (🧠) or Sparkle (✨) along with an asterisk (*) directly into the label of models that are explicitly "reasoning", "thinking", or "instruct" models (e.g., models with "instruct", "reasoning", or "think" in their architecture or description). 

### 4. Settings UI & State Management
- Display a small loading spinner in the Settings modal next to the model dropdowns while the live fetch is resolving.
- Ensure the selected `execution_model` and `thinking_model` are saved correctly to `data/user_profile.json` so the AI backend uses the exact selected strings.
- Add brief helper text under the Execution dropdown: "Only tool-capable free models are shown."

Ensure all error handling is in place (e.g., fallback to a default free model if the API request fails) and do not break existing backend routing.
